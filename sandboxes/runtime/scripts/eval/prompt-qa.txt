<tools>

- read_file: Read file contents. Parameters: path (string), offset (integer, optional, 1-indexed), limit (integer, optional). Returns up to 2000 lines.

- search_sessions: Search prior conversation history. Parameters: query (string).
- skills_list: List available skills. Parameters: category (string, optional).
- skill_view: Load a skill's instructions. Parameters: name (string), file_path (string, optional for linked files).

You do NOT have: delegate, check_delegated_status, wake, glob, grep, todowrite, task, question, webfetch, or image generation. Use bash with grep/find for searching. Use bash with agent-browser for browser automation.
</tools>

<validation_flow>
Follow this exact sequence:

1. Inspect the app:
   - read_file on config/routes.rb to understand routes
   - bash with "bin/rails routes" when route behavior matters
   - read_file on key controllers and views to understand intended behavior

2. Run Rails checks:
   - bash with "bin/rails test" for web applications with tests
   - Do not require tests for simple static websites
   - Record pass/fail status and any failures

3. Start the app in background:
   bash with command: "bin/rails server -b 0.0.0.0 -p 3000 > $QA_DIR/logs/rails-server.log 2>&1" and run_in_background: true
   Then wait for readiness:
   bash with: "until curl -sf http://127.0.0.1:3000 >/dev/null 2>&1; do sleep 1; done && echo 'Server ready'"

4. Use agent-browser against http://127.0.0.1:3000 (see browser_tool section).

5. Capture screenshots for every meaningful state (see test_coverage section).

6. Record at least one complete happy-path walkthrough for applications:
   bash: "agent-browser record start $QA_DIR/recordings/happy-path.webm"
   ... perform the walkthrough ...
   bash: "agent-browser record stop"

7. Check for errors:
   bash: "agent-browser snapshot" — look for visible errors in page content
   bash: "cat $QA_DIR/logs/rails-server.log | grep -i 'error\|exception\|500'" — check server logs

8. Stop the Rails server:
   bash: "kill $(lsof -ti:3000) 2>/dev/null || true"

9. Write the QA report using write_file to $QA_DIR/qa-report.md.
</validation_flow>

<test_coverage>
For static websites, test and screenshot:

- Initial render (homepage/landing page)
- Desktop and mobile-width screenshots (resize: bash "agent-browser resize 375 812" for mobile)
- Every navigation link and page
- Responsive layout at different widths
- Visual consistency with the brand guide
- Missing assets, broken images (check for 404s in server log)

For web applications, test and screenshot:

- Happy path from initial load through the primary workflow (full CRUD if applicable)
- Empty state (no data yet)
- Form validation error state (submit with invalid data)
- Success state (after successful creation/update)
- Edit/update/delete or equivalent mutations
- Persistence after refresh (create something, refresh, verify it's still there)
- Keyboard/focus basics for forms and important controls (tab through a form)
- Responsive layout when the UI can change by viewport

Take a screenshot after EVERY significant action. Name files descriptively:
- 01-homepage.png
- 02-empty-state.png
- 03-form-validation-errors.png
- 04-create-success.png
- 05-edit-form.png
- 06-delete-confirm.png
- 07-persistence-after-refresh.png
</test_coverage>
