1. ARCHITECTURE OVERVIEW
================================================================================

Nango Connect is a three-layer system:

  HOST APP  ←→  FRONTEND SDK (ConnectUI class)  ←→  IFRAME (connect-ui React app)
                                                       ↕
                                                   NANGO API

Layer 1: Host App
  - Imports @nangohq/frontend SDK
  - Creates a ConnectUI instance with a session token
  - Calls connectUI.open() to launch the iframe
  - Receives events (connect, error, close) via callbacks

Layer 2: Frontend SDK (/packages/frontend/lib)
  - ConnectUI class manages iframe lifecycle (create, inject, destroy)
  - Nango class manages OAuth popup windows + WebSocket auth
  - Communicates with iframe via window.postMessage
  - Communicates with host app via callback functions

Layer 3: Connect UI (/packages/connect-ui)
  - Standalone React SPA served from a separate domain (connect.nango.dev)
  - Loaded inside the iframe
  - Communicates with parent via window.postMessage
  - Communicates with Nango API via HTTP (Bearer token auth)

2. IFRAME STRATEGY
================================================================================

2.1 IFRAME CREATION (/packages/frontend/lib/connectUI.ts)
─────────────────────────────────────────────────────────

The ConnectUI class creates and injects an iframe into the host page's DOM:

  class ConnectUI {
    private iframe: HTMLIFrameElement | null = null;

    open() {
      this.iframe = this.createIframe();
      document.body.append(this.iframe);
      document.body.style.overflow = 'hidden';  // Prevent host scroll
      this.setupEventListeners();
    }

    private createIframe(): HTMLIFrameElement {
      const iframe = document.createElement('iframe');
      iframe.src = baseURL + '?' + queryParams;
      iframe.id = 'connect-ui';
      iframe.style.position = 'fixed';
      iframe.style.zIndex = '9999';
      iframe.style.top = '0';
      iframe.style.left = '0';
      iframe.style.right = '0';
      iframe.style.bottom = '0';
      iframe.style.width = '100vw';
      iframe.style.height = '100vh';
      iframe.style.backgroundColor = 'transparent';
      iframe.style.border = 'none';
      return iframe;
    }
  }

Key decisions:
  - position: fixed — overlays entire viewport regardless of scroll
  - z-index: 9999 — sits above all host content
  - 100vw × 100vh — full viewport coverage
  - background: transparent — the React app inside renders its own backdrop
  - border: none — no browser default iframe border
  - document.body.overflow = 'hidden' — prevents host page scrolling under the modal

2.2 IFRAME SIZING MODES
────────────────────────

FULLSCREEN MODE (default):
  - iframe: 100vw × 100vh, fixed position
  - Inside the iframe, Layout component renders a semi-transparent backdrop
  - The actual modal widget is centered: max 500px × 700px
  - CSS: bg-subtle/80 dark:bg-black (backdrop overlay)

EMBEDDED MODE (isEmbedded=true):
  - Used when connect UI is rendered in a constrained container
  - Widget constrained to sm:w-[500px] sm:h-[700px]
  - No backdrop overlay
  - Responsive: full screen on mobile, fixed size on tablet+

AUTH LINK MODE (isAuthLink=true):
  - Direct URL access (not in iframe)
  - Full page layout with padding
  - URL: connect.nango.dev?session_token=ptok_xxx

2.3 URL PARAMETERS PASSED TO IFRAME
────────────────────────────────────

  apiURL                  — Backend API base URL (default: https://api.nango.dev)
  detectClosedAuthWindow  — Boolean, detect if OAuth popup was closed early
  lang                    — Language code (en, fr, de, es)
  theme                   — Theme override (light, dark, system)
  session_token           — Session token (fallback if not sent via postMessage)
  embedded=true           — Rendered inside constrained container
  preview=true            — Preview mode (skips telemetry, for dev/testing)

3. IFRAME COMMUNICATION (PostMessage Protocol)
================================================================================

3.1 MESSAGE TYPES (/packages/frontend/lib/types.ts)
────────────────────────────────────────────────────

PARENT → IFRAME:
  { type: 'session_token', sessionToken: string }
    — Delivers the session token after iframe signals ready

  { type: 'settings_changed', payload: ConnectUISettings }
    — Dynamic settings update (theme changes, etc.)

IFRAME → PARENT:
  { type: 'ready' }
    — iframe has loaded and is ready to receive the session token

  { type: 'close' }
    — User closed the modal (X button, click outside, ESC key)

  { type: 'connect', payload: { providerConfigKey, connectionId, isPending } }
    — Successfully connected a provider

  { type: 'error', payload: { errorType: AuthErrorType, errorMessage: string } }
    — Authentication failed

  { type: 'settings_changed', payload: ConnectUISettings }
    — Settings changed within iframe

3.2 COMMUNICATION FLOW
──────────────────────

  1. Host creates iframe, injects into DOM
  2. iframe React app boots, Home component mounts
  3. iframe sends { type: 'ready' } to parent
  4. Parent receives 'ready', sends { type: 'session_token', sessionToken }
  5. iframe receives token, calls GET /connect/session with Bearer auth
  6. Server validates token, returns session data (allowed integrations, etc.)
  7. iframe navigates to /integrations, shows provider list

  ... user selects provider, enters credentials ...

  8. iframe calls nango.auth() or nango.create() to validate credentials
  9. On success: iframe sends { type: 'connect', payload: {...} } to parent
  10. Parent receives 'connect' event, fires onEvent callback
  11. iframe sends { type: 'close' } on user dismissal
  12. Parent removes iframe from DOM, restores body scroll

3.3 EVENT LISTENER SETUP (/packages/frontend/lib/connectUI.ts)
──────────────────────────────────────────────────────────────

  private setupEventListeners() {
    window.addEventListener('message', this.handleMessage);
  }

  private handleMessage = (event: MessageEvent) => {
    // SECURITY: Origin validation
    if (event.origin !== new URL(this.baseURL).origin) return;

    const data = event.data as ConnectUIEvent;

    switch (data.type) {
      case 'ready':
        // Send session token to iframe
        this.iframe?.contentWindow?.postMessage(
          { type: 'session_token', sessionToken: this.sessionToken },
          this.baseURL
        );
        break;
      case 'close':
        this.close();
        this.onEvent?.(data);
        break;
      case 'connect':
      case 'error':
        this.onEvent?.(data);
        break;
    }
  }

3.4 IFRAME-SIDE LISTENER (/packages/connect-ui/src/lib/events.ts)
────────────────────────────────────────────────────────────────

  // Send message to parent
  export function triggerClose() {
    window.parent.postMessage({ type: 'close' }, '*');
  }

  export function triggerConnect(payload: AuthResult) {
    window.parent.postMessage({ type: 'connect', payload }, '*');
  }

  export function triggerError(payload: { errorType, errorMessage }) {
    window.parent.postMessage({ type: 'error', payload }, '*');
  }

  export function triggerReady() {
    window.parent.postMessage({ type: 'ready' }, '*');
  }

5. SECURITY STRATEGIES
================================================================================

5.1 SESSION TOKEN ARCHITECTURE
──────────────────────────────

  Token format: ptok_<random> (validated by Zod schema)
  Token TTL: 30 minutes (configurable, enforced by keystore)
  Token scope: tied to specific account + environment + end_user

  Session creation (server-side):
    POST /connect/sessions
    Body: {
      end_user: { id, email, display_name },
      organization: { id, display_name },
      allowed_integrations: ['github', 'slack'],
      integrations_config_defaults: { ... },
      tags: { ... }
    }
    Response: { token: 'ptok_xxx', expires_at, connect_link }

  The token is:
    - Created server-side by the HOST APP's backend (not the browser)
    - Passed to the frontend SDK
    - Delivered to the iframe via postMessage (not URL for security)
    - Used as Bearer token for all iframe → API requests
    - Scoped to specific allowed_integrations
    - Time-limited (30 min TTL)

5.2 ORIGIN VALIDATION
─────────────────────

  Parent validates iframe messages:
    if (event.origin !== new URL(this.baseURL).origin) return;

  This ensures only messages from the connect.nango.dev domain are processed.
  Prevents other iframes or malicious scripts from injecting fake events.

  iframe sends messages with wildcard origin ('*') but the parent
  validates the event.origin on receipt. This is the correct pattern:
  the RECEIVER validates, not the sender.

5.3 TOKEN DELIVERY SECURITY
───────────────────────────

  Primary: postMessage (after iframe signals 'ready')
    - Token never appears in URL
    - Token never appears in browser history
    - Token is delivered directly to iframe window context

  Fallback: URL query parameter (session_token=...)
    - Used for "auth link" mode (direct URL access, no parent frame)
    - Less secure — token visible in URL, browser history, server logs
    - Used only for shareable auth links, not iframe mode

5.4 CREDENTIAL HANDLING
───────────────────────

  - API keys entered in the iframe are sent directly to the Nango API
  - Never passed through postMessage to the parent
  - Never stored in the iframe's localStorage or sessionStorage
  - Transmitted over HTTPS to the API endpoint
  - The host app only receives { providerConfigKey, connectionId }
    — never the raw credential

5.5 CORS CONFIGURATION
──────────────────────

  Server routes use explicit CORS:
    origin: connectUrl (e.g., https://connect.nango.dev)
    credentials: false

  Only the connect UI domain can call connect-specific API endpoints.
  The host app's domain is NOT allowed to call these endpoints directly.

5.6 CONTENT SECURITY POLICY
───────────────────────────

  The iframe is loaded from a separate domain (connect.nango.dev),
  so the host app's CSP must allow:
    frame-src: https://connect.nango.dev

  The connect UI's own CSP allows:
    connect-src: https://api.nango.dev (API calls)
    script-src: 'self' (no external scripts)

5.7 SCOPE LIMITATION
────────────────────

  Sessions are scoped:
    - allowed_integrations: only these providers shown in the UI
    - integrations_config_defaults: pre-filled config per provider
    - end_user: session tied to specific user identity
    - account + environment: multi-tenant isolation

  A token for "github only" cannot be used to connect "slack".

7.2 IFRAME INITIALIZATION
─────────────────────────

  1. Host app frontend creates ConnectUI with token
  2. ConnectUI.open() creates iframe pointing to connect.nango.dev
  3. React app boots inside iframe
  4. Home component mounts, sends { type: 'ready' } to parent
  5. Parent receives 'ready', sends { type: 'session_token', sessionToken }
  6. Home component receives token, stores in Zustand
  7. Home calls GET /connect/session with Bearer token
  8. Server validates token (TTL check, scope check)
  9. Server returns ConnectSessionOutput:
     {
       endUser: { id, email, display_name },
       connectUISettings: { ... },
       allowed_integrations: ['github', 'slack'],
       integrations_config_defaults: { ... },
       isReconnecting: false,
       overrides: { ... }
     }
  10. App navigates to /integrations

7.3 PROVIDER SELECTION
──────────────────────

  1. IntegrationsList renders allowed integrations
  2. Each integration card shows: icon, name, description
  3. User clicks integration
  4. App fetches GET /providers/:provider for credential schema
  5. Provider metadata includes:
     - auth_mode: 'OAUTH2' | 'API_KEY' | 'BASIC' | 'JWT' | ...
     - credentials: Zod schema for form generation
     - docs_url: link to provider console
  6. Navigates to /go with provider in Zustand store

7.4 CREDENTIAL COLLECTION (/go route)
──────────────────────────────────────

  For API Key auth:
    1. Form dynamically generated from provider's credential schema
    2. User enters API key
    3. On submit, calls nango.create(providerConfigKey, connectionId, {
         credentials: { apiKey: 'sk-xxx' }
       })
    4. SDK sends POST /api-auth/api-key/:providerConfigKey
    5. Server validates credentials against provider API
    6. On success: { providerConfigKey, connectionId, isPending: false }

  For OAuth auth:
    1. Shows "Connect with {Provider}" button
    2. On click, calls nango.auth(providerConfigKey, connectionId)
    3. SDK opens OAuth popup via AuthorizationModal
    4. SDK establishes WebSocket to Nango server
    5. Server sends connection_ack with ws_client_id
    6. Popup navigates to OAuth provider with ws_client_id
    7. User authorizes in popup
    8. OAuth callback hits Nango server
    9. Server sends 'success' via WebSocket
    10. SDK resolves auth() promise
    11. Popup auto-closes

  For Basic auth:
    1. Form shows username + password fields
    2. On submit, calls nango.create() with { username, password }
    3. POST /api-auth/basic/:providerConfigKey

7.5 CONNECTION RESULT
─────────────────────

  On success:
    - iframe sends postMessage { type: 'connect', payload: { providerConfigKey, connectionId, isPending } }
    - Parent's onEvent callback fires
    - If isPending: connection being tested, may update later
    - If not isPending: connection is active and ready

  On error:
    - iframe sends postMessage { type: 'error', payload: { errorType, errorMessage } }
    - Error types: 'invalid_credentials', 'connection_test_failed', 'unknown'
    - UI shows error screen with retry option

12. WEBSOCKET / REAL-TIME (OAuth Popups Only)

12.1 AUTHORIZATION MODAL (/packages/frontend/lib/authModal.ts)
─────────────────────────────────────────────────────────────

  Used ONLY for OAuth flows that require popup windows.
  NOT used for API key / basic auth flows.

  class AuthorizationModal {
    private swClient: WebSocket;

    constructor({ webSocketUrl, successHandler, errorHandler }) {
      this.swClient = new WebSocket(webSocketUrl);

      this.swClient.onmessage = (message) => {
        const data = JSON.parse(message.data);

        switch (data.message_type) {
          case 'connection_ack':
            // Server acknowledged WebSocket connection
            // Now safe to open the OAuth popup
            this.baseURL.searchParams.set('ws_client_id', data.ws_client_id);
            this.open();  // window.open(oauthURL)
            break;

          case 'success':
            successHandler(data);
            this.swClient.close();
            break;

          case 'error':
            errorHandler(data.error_type, data.error_desc);
            this.swClient.close();
            break;
        }
      };
    }
  }

12.2 OAUTH POPUP FLOW
─────────────────────

  1. nango.auth('github') called
  2. SDK opens WebSocket to wss://api.nango.dev/
  3. Server sends { message_type: 'connection_ack', ws_client_id: 'xxx' }
  4. SDK opens popup: window.open(oauthURL + '?ws_client_id=xxx')
  5. User authorizes in popup window
  6. OAuth provider redirects to Nango callback URL
  7. Nango server processes callback, stores tokens
  8. Nango server sends { message_type: 'success', ... } over WebSocket
  9. SDK receives success, resolves promise
  10. Popup auto-closes

  Why WebSocket instead of polling?
    - Instant notification when OAuth completes
    - No polling overhead
    - Works even if popup redirects to different domain
    - Server pushes result, client doesn't need to poll

15. KEY ARCHITECTURAL DECISIONS
================================================================================

  1. SEPARATE DOMAIN FOR CONNECT UI
     - Hosted on connect.nango.dev, separate from api.nango.dev
     - Isolates credential handling from host app
     - Host app never has access to raw credentials
     - CORS restricts API access to connect domain only

  2. IFRAME OVER WEB COMPONENT
     - Full DOM isolation (no CSS leakage from host)
     - Security boundary (separate origin)
     - Cannot be tampered with by host app JS
     - Downside: heavier, requires postMessage communication

  3. TOKEN DELIVERY VIA POSTMESSAGE (not URL)
     - Token not in URL = not in browser history
     - Token not in server access logs
     - Token not in Referer headers
     - Fallback URL param only for auth link mode

  4. SERVER-SIDE SESSION CREATION
     - Host app backend creates session with secret key
     - Frontend never has the Nango secret key
     - Session scoped to specific user + allowed integrations
     - Time-limited (30 min default)

  5. FULLSCREEN IFRAME WITH TRANSPARENT BG
     - iframe covers entire viewport
     - React app renders its own backdrop overlay
     - Allows custom backdrop styling
     - Click-outside detection handled inside iframe

  6. WEBSOCKET FOR OAUTH COMPLETION
     - Instant notification, no polling
     - Reliable even with cross-domain redirects
     - Server pushes result to client

  7. SENDBEACON FOR TELEMETRY
     - navigator.sendBeacon() guarantees delivery on page unload
     - Non-blocking, doesn't delay user interaction
     - Fire-and-forget pattern
